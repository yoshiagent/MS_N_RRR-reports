🟡 這是設計預覽頁，僅供 review。 classify_n_state 尚未改寫，n_shape.html / box_structure.html 仍跑舊 7 狀態邏輯。 本頁顯示新狀態定義、條件、操作含意、與舊→新映射。

📐 設計原則

2D 正交：位置 × 方向
• 3 位置：箱上 (above SH) / 箱內 (between SL & SH) / 箱下 (below SL)
• 3 方向：向上 (近 3 根 close 斜率 > 0) / 向下 (斜率 < 0) / 特殊（假突破/盤整/破底翻）
• 3 修飾詞：突破確認中、跌破確認中、壓縮整理（不破壞 9 宮對稱）
• 1 unknown：資料不足（獨立）

🗺 9 宮格主結構

向上 →
向下 ↓
特殊 ⚡
箱上 above
🟢箱上推進above_up
條件
close > 最近 confirmed SH，且近 3 根 close 斜率 > 0
說明突破前 SH 後持續向上推進，N 形延續最強訊號
🟢 續抱 / 加碼（趨勢確認）
規範 Part 2 ② N 形延續舊：running
🟡箱上回測above_down
條件
close > 最近 confirmed SH，但近 3 根 close 斜率 < 0
說明曾突破 SH，現回測中。若 SH 守住 → 健康回檔；若跌回箱內 → 假突破
🟡 警戒（測 SH 是否變支撐）
規範 Part 2 ② 延伸（健康回檔）舊：running
⚠箱上假突破above_fake
條件
曾 close > SH，但近 N 根（預設 5）K 棒內 close 又跌回箱內
說明假突破：突破 SH 後快速被打回，主力可能在出貨。比 broken 更早期的警示訊號
🔴 立即減碼（突破失敗）
規範新增（實務常見）舊：breaking
箱內 in
🔵箱內向上inbox_up
條件
SL ≤ close ≤ SH，且近 3 根 close 斜率 > 0
說明箱內向上測試 SH，若突破則進入 N 形延續；若失敗則回箱內整理
🔵 觀察突破，預備進場
規範 Part 2 ① N 形觀察（偏多）舊：watching
🟠箱內向下inbox_down
條件
SL ≤ close ≤ SH，且近 3 根 close 斜率 < 0
說明箱內向下測試 SL，若跌破則進入破壞區；若守住則回箱內整理
🟠 警戒 SL，預備減碼
規範 Part 2 ① N 形觀察（偏空）舊：breaking
⚪箱內盤整inbox_side
條件
SL ≤ close ≤ SH，且近 5 根 close 範圍 ≤ ATR × 1.5（橫盤）
說明箱內橫盤，無明確方向。若伴隨量縮 + 接近箱頂 → 升級為「壓縮整理」修飾詞
⚪ 等待方向，不動作
規範 Part 2 ① N 形觀察（中性）舊：watching
箱下 below
🔄箱下反彈below_up
條件
close < 最近 confirmed SL，但近 3 根 close 斜率 > 0
說明跌破 SL 後嘗試反彈，若成功站回 SL → 進入「箱下破底翻」；若失敗 → 回到向下持續
🔄 觀察反彈強度（破底翻機會）
規範 Part 2 ③ 破底翻進行中舊：reversing
🔴箱下持續below_down
條件
close < 最近 confirmed SL，且近 3 根 close 斜率 < 0
說明跌破 SL 後持續創低，N 形已破壞且無反彈跡象
🔴 停損 / 退出
規範 Part 2 ③ N 形破壞舊：broken
⚡箱下破底翻below_recovery
條件
曾跌破 SL，但 close 已站回 SL 之上（或形成新 HH + HL）
說明破底翻成功 — 跌破後反轉站回，可能是底部反轉訊號。對應規範 ④ 重回 N 形
⚡ 機會（謹慎進場 + 設停損）
規範 Part 2 ④ 重回 N 形舊：resumed_n

⚡ 3 修飾詞（角標 / tooltip 顯示）

⏳突破確認中breakout_pending
條件
close 剛突破 SH，但未滿確認窗（預設 3 根 K 棒）
說明突破與假突破的過渡狀態，最高警覺期
⏳ 先觀察 — 等 3 根確認是真突破還是假突破
套用於：above_up、above_down
⏳跌破確認中breakdown_pending
條件
close 剛跌破 SL，但未滿確認窗（預設 3 根 K 棒）
說明跌破與破底翻的過渡狀態，最高警覺期
⏳ 先觀察 — 等 3 根確認是真跌破還是破底翻
套用於：below_down
⚡壓縮整理compression
條件
箱內盤整 + 量縮 + 接近 SH 上半部
說明規範特別提到的概念 — 量價收斂於高檔，多頭壓縮約 80% 突破上行
⚡ 預備進場 — 高機率即將突破
套用於：inbox_side

🚩 用戶決策清單（請 review 後回覆）

  1. 9 + 3 + 1 結構合理嗎？
    有沒有想新增/移除/合併的狀態？特別是 above_down 與 above_fake 的區隔是否清楚？
  2. 條件閾值合理嗎？
    方向斜率用近 3 根、盤整用近 5 根 + ATR×1.5、假突破窗 5 根 — 太嚴/太鬆？
  3. 命名格式：目前同時顯示 above_up（key）+ 「箱上推進」（label）。
    ⭐ 推薦：報告主顯示中文標籤，tooltip 內展示 key 與條件
  4. 修飾詞顯示：
    (a) tooltip 內標註 (b) 主標籤旁加角標 ⏳/⚡ (c) 獨立 pill
    ⭐ 推薦：(a) tooltip + (b) 主標籤加角標（雙重保險）
  5. 修飾詞優先序：當多個修飾詞同時觸發（如 compression + breakout_pending）， 顯示哪個？建議 breakout/breakdown > compression。
  6. 是否要顯示舊→新映射過渡標籤？
    例如：「⏳ 箱上推進（原 running）」過渡期顯示 1-2 個月。
  7. 盤整 vs 箱內向上/向下的優先序：
    若斜率近乎 0 + 範圍小於 ATR×1.5 → 算盤整？
    若斜率正但仍在窄範圍內 → 算 inbox_up 還是 inbox_side？
    ⭐ 推薦：先判盤整 → 否則才看斜率
