第 1 輪 — codex 幾乎全盤同意：①〜⑤ 全部認同 Claude 方向；並補：n_issuer(投信家數) 要與 n_fund(檔數) 分開存（同投信多檔 ETF 不算獨立共識）、三層儲存 schema、forward-test 事件定義。另反提一個 mv_weight_proxy。

第 2 輪 — Claude 再壓 2 點，codex 認錯並反饋更好的：
• A. 打掉 mv_weight_proxy：Claude 指出 (股數×股價)/Σ(股數×股價) 本質就等於 emega 已給的 reported_weight，一樣隨股價波動、沒隔離主動加碼。codex 認錯，定案：active_money = Δreported_weight − price_drift（用自有價格庫扣股價漂移）為主、股數佔比變化為輔，兩者同號＝高信心。
• B. 壓縮過度 schema：Claude 有完整日線價格庫，故報酬/對照組/事件後表現全部事後可重算，只有「emega 每日共識訊號本身」不可重建 → schema 只要 2 層。codex 同意，但正確反駁：有幾個 provenance 欄位（尤其 available_at 資料幾點可得、signal_version、etf_universe_version、fund 納入/排除 flag）價格庫補不回、必須 day-1 存。
③ 收斂後的 v1 定案（已採納）
一句話：每日存 emega 原始持股 snapshot + point-in-time 共識訊號表；共識用 n_issuer / n_fund + reported_weight 聚合；主動調倉存 股數佔比變化 + active_money（扣股價漂移）；選股用 raw rank 當候選池/gating（不做 AUM、不進 alpha 權重）；績效與對照組全部事後由價格庫動態回算；報告 raw 為主 + 產業中性版只做歸因。

主排序：跨投信共識廣度 n_issuer > n_fund > reported_weight 聚合

```text
扣掉股價漲跌後，這檔 ETF 是否主動提高該股資金權重？
```

所以我的定奪是：

```text
v1 必存兩個：
delta_share_count_ratio
active_money

v1 主排序/主解讀用 active_money
delta_share_count_ratio 作為交叉驗證與異常偵測
```

最不容易誤判的用法是看兩者方向是否一致：

```text
active_money > 0 且 delta_share_count_ratio > 0：高信心主動加碼
active_money > 0 但 delta_share_count_ratio <= 0：可能是資金權重微調，需小心
active_money <= 0 但 delta_share_count_ratio > 0：可能偏張數配置或低價股效應
兩者都 < 0：高信心降碼
```

要注意 B 的幾個技術細節：

- 用 `t-1` 的 reported weight 當基準，不要用 `t`。
- portfolio return 要用 ETF 持股內部估算：`sum(weight_j,t-1 * ret_j)`。
- 新增持股沒有 `weight_t-1`，active_money 可直接視為 `reported_weight_t` 或標記為 `new_position = true`。
- 刪除持股若 emega 當日不列出，要靠 yesterday snapshot 補出 `weight_t = 0`，否則會漏掉主動賣出。
- 除權息、停牌、缺價要有 fallback，否則 active_money 會被污染。
- 若 emega 的 `reported_weight` 已有四捨五入，active_money 小數值不要過度解讀，應設最小門檻。

我不會再把 `mv_weight_proxy` 當主動調倉 proxy。最多只在你不信任 emega `reported_weight` 時，用它做資料校驗。

**B. Schema 定奪**

同意你的壓縮。  
在你有完整價格歷史 DuckDB 的前提下，我前面三層 schema 裡的 event/benchmark 大多可以不預存。

真正 day-1 必存的是「未來無法重建的 point-in-time 訊號狀態」。價格、報酬、benchmark、holding period performance 都可以事後重算。

所以 v1 可以只存兩層：

```text
1. raw ETF holding snapshot
2. daily consensus signal
```

event/推薦梯次表不是必要資料層，它可以是查詢結果或 materialized view。

但有幾個欄位我認為現在不存，以後會後悔，因為價格庫補不回來：

```text
signal_version
selection_rule_version
source
raw_hash
ingested_at
asof_date / signal_date
available_at 或 data_cutoff_time
etf_universe_version
issuer mapping
fund inclusion/exclusion flag
stock identity mapping
```

尤其重要的是這幾個。

`signal_version`：  
之後你一定會改公式。沒有版本，forward-test 會混在一起。

`selection_rule_version`：  
即使你不存 event 表，也要知道當時 Top N 是用什麼規則選出來的。可以存在 signal 表，或另外有一張小型 rule registry。

`available_at` / `data_cutoff_time`：  
避免未來不知道這個訊號是在收盤前、收盤後、晚上幾點才可得。這直接影響是否能用隔日開盤/收盤進場。

`etf_universe_version`：  
今天納入哪些主動 ETF，是訊號定義的一部分。未來新增 ETF 後，不能回頭污染舊訊號。

`issuer mapping`：  
`n_issuer` 是核心訊號。若 ETF 名稱、投信名稱之後變動，必須能重建當天口徑。

`fund inclusion/exclusion flag`：  
例如某 ETF 當天資料缺漏、異常、暫停納入。這不是價格庫能補的。

`raw_hash` / `ingested_at`：  
用來查資料污染、重抓、修正。

我對 event 表的結論：

```text
v1 不必存大型 event/forward-test 表。
但必須能從 daily consensus signal 用 signal_version + selection_rule_version 重建當天入選名單。
```

如果你願意多存一個很薄的表，我只會存「selection manifest」，不是績效表：

```text
signal_date
selection_rule_version
stock_code
rank_raw
rank_industry_neutral
selected_bucket
created_at
```

但這不是必須。只要 signal 表完整，查詢時動態算也可以。
