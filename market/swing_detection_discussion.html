SHSL 波段偵測問題與狀態機解法（顧問討論）

顧問討論紀實 ｜ Claude × codex ｜ 孤立候選 → 方向狀態機（ATR ZigZag） ｜ ✅ 已上線
✅ 狀態：已實作上線　｜　find_swings_3stage 已由「兩條候選並行」重構為「方向狀態機（ATR ZigZag）」；全市場 663/663 時段嚴格交替、0 非交替。3037 日線孤立候選 SL（916）已消除。
SHSL 波段偵測「孤立候選」問題 — 顧問討論紀實　Claude × codex 顧問　｜　問題回報 → 根因定位 → 顧問裁決 → 實作（2026-06-28）
CIO 看 欣興 3037 的 B5（SHSL Swing 結構）日線卡，發現一個 SL 916 的綠圈孤零零黏在前一個 SL（811）後面、中間「缺一個高點」。順著查，先抓到回檔箱取法的 bug，再往下挖出 SHSL 核心 swing 偵測的設計缺陷，諮詢 codex 後重構成方向狀態機。
① 問題現象（CIO 回報）
欣興 3037 日線 confirmed 序列顯示成 … SH 1130 → SL 811 →（孤立）候選 SL 916 → 候選 SH 1090，出現兩個連續低點、中間缺一個高，916 的綠圈看起來像孤兒。
同時回檔箱也怪：箱頂被畫在 972（一個夾在主峰 1130 與底部 811 之間的次高），而不是真正啟動回檔的波段峰 1130 →「box 應繪製在確認回檔波」。
3037 日線價格路徑（實際 K 線）說明
06-11 低 811（confirmed SL）前一個確認低點
06-22/23 反彈到峰 ~1075（06-23 高 1075）波段高（但未被登錄成 SH）
06-24 回檔低 9161 根當沖洗盤的低點
06-25/26 再創新高 1090（06-26 高 1090）續攻、創新高
② 根因：SH/SL「兩條候選並行追蹤」
舊版 find_swings_3stage 把 SH 候選與 SL 候選「各跑各的、互不溝通」。於是「確認 SL 811 之後」，SH 那條在追 1090，SL 那條同時又獨立養出一條候選 SL 916——兩條候選平行存在 → 結構不交替、出現孤立候選。
同理，下跌途中的次高 972@06-09 也被誤確認成 SH（夾在 1130 與 811 之間），污染了回檔箱取法（箱頂取「最近 SH」就取到 972 而非真峰 1130）。
一句話：問題不在某個門檻，而在「兩條候選並行」這個結構本身。
③ 兩個修法（Claude 提出）
修法做法對 3037 的結果取捨
A 嚴格交替依最後確認側單向追蹤：確認 SL 後只追候選 SH、確認 SH 後只追候選 SLSL811 → 候選 SH1090（916/1075 都吸收）乾淨、永遠交替；但可能吞掉中繼回檔
B 結清峰谷對新高欲覆蓋待確認峰時，若峰後已有有效谷，先把（峰→谷）結清成一組 SH/SL 再起新峰SL811 → SH1075 → SL916 → 候選 SH1090保留更多波段；強趨勢中可能切太碎
因為是改「全系統最核心」的 swing 偵測（box／世代／n_shape／選股全吃它），blast radius 大，決定先諮詢 codex 顧問再動手（符合「重要演算法決策先過顧問」的工作慣例）。
④ codex 顧問裁決（GPT-5.5）
結論：不要用 A 當 core；B 語義較對但要加護欄；最佳解是改成標準「ATR/deviation ZigZag 狀態機」。
議題codex 觀點
A 當 core 偵測器❌ 不建議：A 會把『已達 ATR 意義的中繼回檔』整段吸收掉，對風控/換股/停損可能誤導；A 只適合當『大波段簡化視圖／後處理 filter』
A 的 edge case過度延遲、吞中繼波段；V 轉/跳空/長盤整都可能只留最後極值；對稱漂亮但資訊損失嚴重
B 會不會碎片化會，強趨勢中尤甚；需加：谷也要 ≥min_bars、用 close 確認、min%/ATR、嚴格順序、谷未達確認前只更新峰不結清
最佳解（第三條）✅ 標準 ATR ZigZag 狀態機：只追單一方向極值，pivot 由『反向幅度達門檻』觸發確認（而非並行養第二條候選線）→ 自然保證交替、又不吞有效回檔
⑤ 決策：三時段統一「方向狀態機」
CIO 一度提「日線用 A、週月線用 B」。評估後否決此分法：
① 方向與 codex 風險分析相反——A 最傷的正是日線（最需要看到回檔的層級）；
② 日線 A／週月 B 混用會破壞「三時段對齊」（同一段行情被不同演算法切出不同結構）。
改採：三時段統一狀態機；「日線要更乾淨、週月可更敏感」用門檻（ATR 倍數／min_bars）調，而非換演算法。既達乾淨、又不犧牲正確性與一致性。

關鍵洞見：3037 的 1075→916 其實是1 根 K 的當沖洗盤（close 從沒跌破峰底、回檔只持續 1 天 < min_bars 3），本來就不該算波段。所以正確的狀態機自然得到 SL811 → 候選 SH1090——同時滿足 CIO 的「嚴格交替」直覺與 codex 的「別丟有效回檔」。
⑥ 實作（find_swings_3stage 重構）
依「最後確認的那一側」單向追蹤：確認 SL → 進「找高（只追候選 SH）」；確認 SH → 進「找低（只追候選 SL）」。確認某側時用「反向極值」（峰後最低谷／谷後最高峰）種下對側候選 → 不遺失真正的反向波段；起手 bootstrap 兩側並追、先確認者定向。
SH-01／02／03 確認條件完全不變（close 破峰底 + 回檔 ≥ATR + ≥min_bars）。差別只在「不再並行養兩條候選」。
程式：src/shared/swing_core.py:find_swings_3stage；只有 shsl_agent 生產 swing，box_structure／n_shape 只讀 FS（R6）。
⑦ 驗證結果（全市場 + 重點股）
檢查項結果
時段嚴格交替✅ 663/663；非交替 0
單側候選不變式✅ 成立（僅 41 檔月線天生 bootstrap，舊版也 0 confirmed，非回歸）
3037 日線✅ SH1130 → SL811 → 候選 SH1090（916／972 皆消失）；box=1130→811；wave=REBOUND
8046 日線✅ 吸收雜訊 914/862（後被連續更高破壞、非真峰），結構更乾淨
重跑鏈shsl_agent → box_structure → n_shape → 重建報告 → 部署；smoke 19/19、audit 通過
🎯 收穫　CIO 的「嚴格交替」直覺其實命中了根因（並行追蹤）；codex 的價值在於指出「別用會丟有效回檔的版本當 core、改用標準 ZigZag 狀態機」。最後兩者收斂：方向狀態機=既乾淨、又不丟真波段。同一支修正連帶解掉了回檔箱取法 bug（972 不再被確認）。
