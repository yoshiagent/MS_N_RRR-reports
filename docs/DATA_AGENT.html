Data Agent 規格

對應實作：C:\CludeHome\stockHisData\stock_data\ 規格類別：基礎設施 Agent（與決策層 Agent 不同，不參與市場判斷）

任務

統一管理所有台股歷史資料，作為跨 session / 跨專案的單一資料來源。 負責：

  • 從外部資料源（yfinance、TWSE、TPEX、TAIFEX）抓取資料
  • 處理（編碼轉換、欄位正規化、日期對齊、ROC ↔ AD 換算）
  • 增量更新與本地快取
  • 提供讀取 API 與 CLI

否決條件（失敗處理）

情境 行為
個別 fetch 拋例外 log warning，跳過該日 / 該檔，繼續下一個
個股 .TW / .TWO 都失敗 若有本地快取則用快取；否則 raise
TWSE 該日無資料（假日 / 尚未公布） 跳過該日
TPEX OpenAPI 失敗 log warning，不影響 TWSE 部分
編碼錯誤 嘗試 ms950 → big5 → cp950，最後 errors="replace"
本地 CSV 損毀 log warning，視同無資料；重抓

不會做的事（明確界線）

  • 不分析、不評分、不下交易建議
  • 不持有狀態（無 session / DB / 鎖）
  • 不處理盤中即時報價（盤中即時用 XQ DDE，由消費者自行處理）
  • 不爬主力分點、不查借券、不抓基本面（之後再加，需另立規格）
  • 不替任何專案做特定欄位處理（如月線重採樣由消費者呼叫 load_price(timeframe="monthly") 處理）

介面契約（Public API）

任何呼叫端可預期：

  1. 所有 load_* 函式回傳 pandas DataFrame，索引為 DatetimeIndex（無時區）
  2. 缺資料時不靜默回空 — 會 raise RuntimeError 或 KeyError
  3. refresh=False 強制只讀本地，不打網路
  4. 所有 fetch_* 函式皆為冪等增量：重複呼叫安全，重複 fetch 同一日會 overwrite 該日
  5. 所有寫入採 keep="last" — 同日新資料會覆蓋舊資料
  6. 不刪除任何既有資料

3. 踩坑經驗（按重要性排序）

編碼 / 解析

  1. TAIFEX CSV 不是 UTF-8 → UnicodeDecodeError 解：嘗試 ms950 → big5 → cp950，最後 errors="replace"

  2. industry_sub.csv 用 pandas read_csv 直接讀必炸（"Expected N fields, saw M"） 解：用 csv.reader([ln]) 逐行 + 正規表達式找代號 ^\d{4,5}[A-Z]?\.(TW|TWO)$

  3. TPEX OpenAPI 欄位名稱前面有 leading space " Foreign Investors include Mainland..." 不是 "Foreign..." 解：hardcode 精確字串（含前置空白）對應

  4. TWSE T86 row 長度不固定（某些日子部分 row 欄位少於 19） → IndexError 解：if len(row) < max(_T86_IDX.values()) + 1: continue

  5. 數值字串含逗號 "156,969,558" 直接 int() 炸 解：自寫 _parse_num(s) 處理逗號 / 空字串 / -- / 浮點 fallback

Pandas / 型別

  1. date 欄型別混雜 → '<' not supported between instances of 'str' and 'Timestamp' 發生在 append 新 row 與既有 CSV 合併排序時 解：新 row 建構用 pd.Timestamp(d)；read_csv(..., parse_dates=["date"])；寫出 date_format="%Y-%m-%d"

  2. 同日重複寫入造成重複列 解：concat 前先 df = df[df["date"] != new_ts] 去舊

  3. yfinance 新版 MultiIndex columns（即使單一 ticker） 解：df.columns = [c[0] for c in df.columns] 攤平

API 行為

  1. TPEX OpenAPI 忽略 date 參數（永遠回最新） 解：歷史走 RWD endpoint；OpenAPI 只用來補最新一日

  2. TPEX columnNum 與實際 fields 不符（25 vs 24） 解：以 fields array 長度為準

  3. TPEX 歷史 endpoint 接受兩種日期（民國 / AD 都通） 解：統一傳 AD YYYY/MM/DD 即可

  4. 股票代號 .TW vs .TWO 上市 = .TW（2330.TW），上櫃 = .TWO（6488.TWO） 不知道時：先試 .TW 失敗再 .TWO（已實作於 fetch_price）

  5. TWSE T86 當日資料 17:30 後才出 14:30 收盤馬上跑只能拿到 stat = "很抱歉..."

排程 / 流程

  1. 業務日 ≠ 國定假日 _business_days() 只跳週末，沒考慮春節 / 中秋 / 228 / 雙十 / 補班 假日 fetcher 回空 dict 但無 error → 設計上接受

  2. PowerShell 與 git-bash 引號互不相容 Bash here-string 跨平台炸，PowerShell "\"轉義常出錯 解：寫獨立 .py 檔再 python -m，不要在 shell 拼複雜命令

  3. per-symbol loop 抓 T86 / TPEX 是天大浪費 這兩個 endpoint 一次 call 回全市場 1,300 / 900 檔 解：永遠 per-day loop，不要 per-symbol loop

4. 效率心法

做法 為什麼
TWSE / TPEX 一次抓全市場 T86 一次 ~1,300 檔 / TPEX 一次 ~900 檔；per-symbol loop 是 1,300× 慢
TAIFEX 與 TWSE 平行跑 不同網域 / 不同 rate-limit pool，雙 process 並行省一半時間
本地最新 < today - 3 才打網路 _need_refresh() 的 3 自然日 grace；連續週末不會誤觸發
fetch_* 函式增量且冪等 反覆呼叫安全；同日 fetch 兩次只是覆蓋同 row
「指定 symbol 都已有的日期交集」才算 covered 擴大 symbol set 時新檔自動補；老檔不再重抓
throttle 寫死在 fetch 函式內 不要靠呼叫端自律
yfinance 三件套 progress=False, auto_adjust=True, actions=False 必加
大量補抓走背景 250 日 × 1.5s ≈ 7 分鐘；用 run_in_background 別佔 foreground
進度與結果寫 log 檔 背景任務無法 print 觀察，靠 meta/update_log.csv 與 logs/*.txt

DuckDB 操作節

對應實作：stock_data/db.py + stock_data/queries.py DB 檔案：database/stocks.duckdb

為什麼用 DuckDB

擴大到全市場（~2,700 檔）後，原本 pandas + CSV 的 cross-section 與 window 查詢太慢。 DuckDB columnar + vectorized 引擎，分析查詢比 SQLite 快 10-100×。詳見專案根 README 對照表。

8.1 Schema（7 張表 + 4 個索引）

price          (symbol, date, open, high, low, close, volume)              PK (symbol, date), idx date
institutional  (symbol, date, "foreign", trust, dealer, total)              PK (symbol, date), idx date
futures        (date, foreign_tx_oi_net, trust_tx_oi_net, dealer_tx_oi_net) PK date
indices        (name, date, OHLCV)                                          PK (name, date),   idx date
macro          (key,  date, OHLCV)                                          PK (key,  date),   idx date
symbols        (symbol, name, market, industry_group)                       PK symbol
update_log     (ts, source, rows, extra)                                    log-only

關鍵設計： - date 用 DATE 而非 TIMESTAMP（PK 比對時不會被時分秒搞亂） - 同表所有寫入使用 INSERT INTO ... ON CONFLICT (...) DO UPDATE SET ...，確保冪等 - "foreign" 因與 FOREIGN KEY 同字，所有 SQL 都要用雙引號包覆

8.4 Dual-write 機制

fetchers.py 內所有 fetch_* 函式在寫 CSV 後同時 upsert 進 DuckDB：

  • 開關：fetchers.DUAL_WRITE_DB = True/False
  • 失敗安全：DB upsert 失敗只 log.warning，不中斷 CSV 主流程
  • 內部包裝：_db_upsert(fn_name, *args) 處理 try/except

這是讓 DB 從 fetcher 一直保持與 CSV 同步的關鍵。

8.5 踩坑與 DuckDB-specific 陷阱

A. "foreign" 必須引號

-- 錯
SELECT foreign FROM institutional;
-- 對
SELECT "foreign" FROM institutional;

原因：FOREIGN 是 SQL 關鍵字（FOREIGN KEY 用），DuckDB parser 看到沒引號的 foreign 後面跟著 FROM 或型別會困惑。 建議：在所有 SQL 中永遠寫 "foreign"，即使在 column list 上。

B. CAST 不能用 parameter binding

-- 錯（DuckDB Parser Error）
WHERE date > (DATE ? - INTERVAL '10 days')
-- 對
WHERE date > CAST(? AS DATE) - INTERVAL 10 DAY

C. INTERVAL 字面值不能用 parameter

-- 錯
WHERE date > today_date - INTERVAL '? days'
-- 對：在 Python 端組字串
sql = f"WHERE date > today_date - INTERVAL {n_days} DAY"

D. INSERT 時 column 順序要對

df_in = df[all_cols]  # 重排欄位順序與 schema 一致
con.execute(f"INSERT INTO price ({cols_sql}) SELECT {cols_sql} FROM df_in ...")

E. DataFrame 註冊用 con.register

con.register("df_in", df_in)
try:
    con.execute("INSERT INTO t SELECT * FROM df_in ON CONFLICT (...) DO UPDATE SET ...")
finally:
    con.unregister("df_in")  # 解綁避免污染後續 query

F. date 型別處理

DataFrame 的 date 欄要先轉 dt.date（不要保留 datetime64[ns]），否則 ON CONFLICT 比對失敗：

df["date"] = pd.to_datetime(df["date"]).dt.date

G. PRIMARY KEY 衝突處理

所有 upsert 用 ON CONFLICT (key_cols) DO UPDATE SET col1=EXCLUDED.col1, ...。 不要用 INSERT OR REPLACE（PostgreSQL 才支援，DuckDB 雖然能跑但語意有微差）。

H. 多進程寫衝突

DuckDB 像 SQLite 是 single-writer。同時兩個 process 寫會 raise IO error。 解：fetcher 串行跑（默認）；若必須並行，加 retry-with-backoff。

I. read_only 模式提升讀效能

唯讀 query 用 db_connect(read_only=True)，DuckDB 會略過 WAL 處理，多進程也安全：

con = db_connect(read_only=True)

8.6 常用 SQL 範本

全市場最新一日截面

SELECT p.symbol, s.name, p.close, p.volume,
       i."foreign", i.trust, i.dealer
FROM price p
LEFT JOIN symbols s        USING (symbol)
LEFT JOIN institutional i  ON p.symbol = i.symbol AND p.date = i.date
WHERE p.date = (SELECT MAX(date) FROM price);

N 日報酬 ranking

WITH ranked AS (
    SELECT symbol, date, close,
           LAG(close, 5) OVER (PARTITION BY symbol ORDER BY date) AS close_5d
    FROM price
)
SELECT symbol, ROUND((close - close_5d) / close_5d * 100, 2) AS ret_5d
FROM ranked
WHERE date = (SELECT MAX(date) FROM price) AND close_5d IS NOT NULL
ORDER BY ret_5d DESC LIMIT 20;

配對 RS + Z60

WITH base AS (
    SELECT p1.date, LN(p1.close / p2.close) AS rs
    FROM price p1 JOIN price p2 ON p1.date = p2.date
    WHERE p1.symbol = '2382' AND p2.symbol = '2330'
    ORDER BY p1.date
)
SELECT date, rs,
       (rs - AVG(rs) OVER (ORDER BY date ROWS 59 PRECEDING)) /
       NULLIF(STDDEV_POP(rs) OVER (ORDER BY date ROWS 59 PRECEDING), 0) AS z60
FROM base;

三表 ASOF JOIN（對齊不同 cadence 用）

SELECT p.date, p.close, i."foreign", f.foreign_tx_oi_net
FROM price p
ASOF JOIN institutional i ON p.symbol = i.symbol AND p.date >= i.date
ASOF JOIN futures f       ON p.date >= f.date
WHERE p.symbol = '2330'
ORDER BY p.date DESC LIMIT 10;
